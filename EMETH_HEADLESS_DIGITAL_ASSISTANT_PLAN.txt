Emeth Headless Digital Assistant Product Plan
Date: 2026-06-22

Purpose
Emeth should become a useful, non-intelligent Android digital assistant: deterministic, permission-aware, fast, private, and honest. It does not need general AI to be valuable. It needs to reliably understand common commands, know what it can do, complete safe tasks headlessly, prepare risky tasks cleanly, and explain exactly where Android or app limitations require confirmation.

Current capability inventory
Emeth already has a working kernel:
1. HomeScreen sends user text to Planner.process().
2. IntentResolver normalizes text, expands phrases, extracts entities, and maps text to an Intent.
3. SkillRegistry selects a Skill.
4. ExecutionEngine runs the Skill and publishes events.
5. MemoryStore logs commands and app usage.
6. WatcherWorker supports periodic background checks.

What Emeth can do now
- Device info: check battery, storage, RAM, clipboard, and some step-count state.
- Device actions: flashlight, mute, volume surface, brightness/settings surfaces.
- Apps and media: open YouTube, search YouTube, open some YouTube deep links, play a URL, remember last played YouTube URL, open camera, video camera, and voice recorder.
- Productivity: set alarms, timers, stopwatch surface, calendar surfaces, and step watchers.
- Communication: open dialer, SMS surface, contacts search, WhatsApp open/share/pre-send surfaces.
- Files: query MediaStore, rank likely files, list deletion/move candidates, detect obvious duplicates by name and size, and find latest downloaded video for WhatsApp status pre-send.
- Memory: store command/app usage records and answer basic activity queries.
- Conversation: basic greetings, identity, capability fallback, unsupported-command fallback.

What keeps it from being a paid-useful headless assistant
- It cannot yet classify each action by execution mode: true headless, app intent, confirmation-required, accessibility-required, or unsupported.
- Many skills still open screens instead of completing tasks.
- Some skills ignore extracted entities or use placeholders: web search query, contacts, timers, notes/reminders/calendar details, generic app open.
- Confidence is fake; ParsedCommand currently returns 1.0 even for weak matches.
- Partial prompts do not preserve pending task state, so follow-up replies are treated as new commands.
- There is no permission and capability recovery model: Emeth should know what permission or service is missing and guide the user.
- WhatsApp, YouTube like/comment/subscribe, unread chat state, status posting, and many third-party app actions are not public Android app-intent actions. They need Accessibility, notification listener, app-specific APIs, or user confirmation.
- File delete/move work is selection-oriented but does not yet execute Android MediaStore delete/write requests after user selection.
- Memory is mostly a log, not a context engine for preferences, recent targets, or task continuation.

Definition of "true headless" for Emeth
- Headless means no visible app launch and no final user tap required.
- App-intent assisted means Android or another app opens a screen with data prefilled.
- Confirmation-required means Emeth can prepare the exact action, but Android or product safety requires the user to confirm.
- Accessibility-required means Emeth can theoretically operate another app's UI if an Accessibility service is built and explicitly enabled.
- Unsupported means public Android APIs and app intents do not expose the action.

Paid-useful product direction
Emeth should become excellent at a limited set of practical workflows:
1. File command center: find files by fuzzy name, type, recency, source, and duplicate groups; safely delete/move/share after explicit selection.
2. Communication prep and automation: contacts, SMS, WhatsApp pre-send, status prep, unread-notification summaries, and clear limits for headless third-party messaging.
3. Personal command memory: remember recent files, chats, apps, searches, and user preferences.
4. Device automation: watchers for battery, storage, steps, time, location-ready future hooks, and app/package events.
5. Intent router: understand, clarify, resume, execute, or refuse based on capability and risk.
6. Permission cockpit: show what is enabled, what is missing, and what new powers each permission unlocks.

Implementation plan
Step 1: Capability ledger and headless contract
- Add a central CapabilityRegistry that maps intents to execution mode, risk level, required permission/service, and user-facing limit.
- Add a capability query skill so Emeth can answer "what can you do headlessly?" honestly.
- Use this registry as the source of truth for future UI chips, planner checks, and safety gates.

Step 2: Real confidence and clarification
- Replace Intent-only matching with IntentMatch(intent, score, reasons).
- Set ParsedCommand.confidence from the score.
- Clarify missing fields for contact, file, destination, message, query, timer duration, alarm time, and calendar details.

Step 3: Pending command continuation
- Store the last Partial command with missing slots.
- Merge the user's next reply into the pending command.
- Add expiry and cancel commands.

Step 4: Entity-driven core skills
- Make web search use the extracted query.
- Make generic app open resolve installed apps and aliases.
- Make contacts/call/SMS/WhatsApp use extracted contact and message data.
- Make timers, reminders, notes, and calendar use extracted content instead of placeholders.

Step 5: File manager execution
- Add user-selection commands for file candidates.
- Use Android MediaStore create/delete/write APIs where possible.
- Add duplicate safety: keep newest, keep largest, never delete all copies, and require confirmation for destructive batches.

Step 6: Permission cockpit
- Track contacts, calendar, media, notifications, activity recognition, notification listener, accessibility, and manage-files status.
- Return recovery actions when a skill fails because capability is missing.

Step 7: Optional Accessibility service
- Add explicit user-enabled Accessibility only for actions public intents cannot complete.
- Build guarded UI automation recipes for supported apps.
- Keep risky actions confirmation-first unless the user explicitly opts into a trusted automation rule.

Step 8: Useful background automation
- Expand WatcherWorker into battery/storage/file-download/app-event watchers.
- Let users create simple deterministic rules such as "when a PDF downloads, show it in Emeth" or "when storage is low, list cleanup candidates."

Step 9: Product polish
- Add a capability dashboard, pending-task UI, candidate selection UI, and concise activity log.
- Package common workflows as command cards: files, WhatsApp prep, YouTube, device cleanup, reminders, and watchers.

Air OS shared spine phase implemented
- Added a signature-protected ContentProvider at content://com.airos.spine.
- Added one shared identity scheme:
  - Stable IDs represent the same entity across every Air OS node.
  - Unique IDs represent memories and behavioral events.
- Added shared persistent tables for:
  - Nodes and app identities
  - Memories
  - UI maps
  - Access mechanisms
  - Behavioral events
- Emeth registers as the first assistant node during startup.
- Emeth publishes its navigation map and Android access mechanisms into the spine.
- Every successful or failed command is written to shared memory and the behavioral stream.
- Accessibility publishes a throttled active-screen UI map with stable node IDs.
- Access is restricted to apps signed with the same Air OS signing certificate.
